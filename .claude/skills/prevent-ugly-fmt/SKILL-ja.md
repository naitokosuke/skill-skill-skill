# prevent-ugly-fmt

## 原則

Formatter の出力が気持ち悪いとき、原因はソース構造が formatter の規則と衝突していること

formatter 側を抑制 (ignore コメント等) するのではなく、ソース側を最小に書き換えて formatter が自然にきれいな出力を返すようにする

問題を別の媒体 (CSS pseudo-element、画像、注釈等) に逃がすのも避ける

## 手順

- 出力の何が気持ち悪いかを形 (パターン) として言語化する
  - たとえば「閉じタグの `>` だけが次行にある」
  - たとえば「三項の `?` と `:` の位置が、対応する条件と揃っていない」
  - たとえば「メソッドチェーンが意味の切れ目と違うところで折られる」
- その形を生む最小単位をソースで特定する
  - たいていは 1 つの式・1 つの要素・1 行に隣接した複数の構造
- その単位を同じ意味を保ったまま構造的に書き換える
  - よく効く操作
    - 隣接物の分離 (改行を入れる、独立行に置く)
    - 中間変数 / 中間要素への抽出
    - 複合式を分解 (条件式 → if/else、三項ネスト → 早期 return)
    - 親側で空白制御を引き受ける (CSS の `display: flex` で子間空白を消す等、構造で意味的等価を担保)
- formatter を実行して (プロジェクトが定めたラッパー、何であれ) 出力を目視
  - 再発したら別の書き換えへ

## 検証

formatter は変更ごとに走らせる

1 回通っても、近くの構造を増減すると再発しうる

## 紛らわしい代替案

着手前に魅力的に見えるが、それぞれ別のコストに置き換わる

- `// prettier-ignore` / `// fmt: skip` 等の ignore コメント
  - プロジェクト規約違反、別の formatter は無視できない、レビュアーが驚く
- 表示文字列を CSS `content` / 画像 / 注釈に逃がす
  - コピー・検索・読み上げの対象から外れ、a11y/UX が劣化する
- データの形 (文字数・配列長等) に依存する見た目調整
  - データが変わった瞬間に崩れる fragile な実装
- 視覚的に等価な別構造で「ごまかす」 (隣接 inline 要素 2 つ等)
  - 同じ formatter 規則で再度崩される
  - 原因 (構造の衝突) が解決していない

## 具体例 — Prettier 系 HTML formatter

入力 `<dt><b>X</b>/total</dt>` が `</b\n>/total` に崩される

原因は Prettier がインライン要素と隣接テキスト間を whitespace-sensitive と判定し、空白を入れたくないため `>` だけを次行に押し出すこと

隣接テキストを独立行に置くことで衝突解消

```html
<dt>
  <span class="count">X</span>
  /total
</dt>
```

閉じタグの `>` が行末で完結し、トレーリングテキストは子テキストノードとして残る

a11y/コピー/検索すべて素のまま機能する

補足として、HTML の whitespace collapse により、新しいレイアウトでは要素とトレーリングテキストの間に半角スペース 1 個が入る

視覚的に詰めたい場合は親に `display: flex` を当てる (flex では子要素間のソース空白は無視される)

## 一般化

formatter ごとに「壊れる形」は違うが、対処は同じ

形を観察 → 構造を書き換え → 再実行
