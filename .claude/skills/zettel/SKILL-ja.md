# zettel（技術知識帳）の作成・更新

`~/zettel/` 配下に、プロジェクトに依存しない技術メモを作成・更新する。

## 指示の解釈

```
$ARGUMENTS
```

ユーザーの指示から以下を特定する:

- 記載するトピック（技術、概念、移行、トラブルシュート等）
- 既存ファイルの更新か、新規作成か
- 関連する調査対象（ブランチ、ファイル、エラー等）

既存ファイルを更新するか新規作成するかは `ls ~/zettel/` で確認する。

## ファイル名規則

- 配置先: `~/zettel/{slug}.md`
- slug は kebab-case の英小文字
- トピックを的確に表す短い名前にする
  - 良い例: `vue-transition-group.md`、`migrate-ghq.md`、`oxlint-raw-transfer-bumpalo.md`
  - 悪い例: `memo.md`、`vue.md`、`2026-05-01.md`

## 最重要原則: プロジェクト非依存

zettel は将来の自分や別プロジェクトでも参照できる汎用的な知識帳である。
プロジェクト固有の情報を書き込まないこと。

### NG な書き方の例

```markdown
`src/app/todos/__features/TodoTrees.vue` で transition-group を使っている
```

```markdown
このプロジェクトの `AfExpandTransition` コンポーネントと組み合わせると...
```

```markdown
MR #3777 で導入した
```

### OK な書き方の例

プロジェクト固有のコンポーネント名やファイルパスは、汎用的な例に置き換える:

```markdown
ツリーの折り畳み開閉時に、子 node を 1 つずつアニメーションさせるリスト用途で使う
```

```vue
<transition-group v-if="folder.isOpened && folder.children?.length" ...>
  <ChildNode v-for="child in folder.children" :key="child.key" />
</transition-group>
```

調査の起点がプロジェクトのコードであっても、得られた知見は「一般化された原理」「他プロジェクトでも通用する例」として書き直す。

## 文書構造

### 構成例

```markdown
# {トピック名}

{1〜2 段落の概要。何の話か、いつ役に立つか}

## {主要セクション}

{原理や仕組みの説明}

## {使い方 / コード例}

{汎用的な最小コード例}

## {応用 / 注意点}

{エッジケース、よくあるハマりどころ}

## 参考

- [{タイトル}]({URL})
```

### 推奨される内容

- 概念の定義と「なぜそうなっているのか」の説明
- 最小限で動く汎用的なコード例
- 公式ドキュメントへのリンク
- バージョン間の破壊的変更（移行作業の文脈で調べた場合）
- 比較表や対応関係（API rename、設定値の対応等）

### 避ける内容

- プロジェクト固有のファイルパス、コンポーネント名、変数名
- 特定の MR / PR / issue / commit への言及
- 「このブランチで」「今回の作業で」のような文脈依存の記述
- 一回限りのデバッグログや作業履歴（それは作業メモであって zettel ではない）

## frontmatter とタグ

すべての zettel ファイルの先頭に YAML frontmatter で `tags` を記載する。

```yaml
---
tags:
  - frontend
  - vue
  - migration
---
```

### タグ選定の手順

1. `~/zettel/_tags.md` を読み、既存タグの定義を確認する
2. カテゴリタグ (`frontend` / `tooling` / `migration` 等) を必ず 1 つ含める
3. 関連する技術タグを必要なだけ付ける
4. 既存タグで表現できない概念があるときに限り、新しいタグを追加する
5. 新しいタグを追加した場合は、`~/zettel/_tags.md` に定義と用途を必ず追記する

### タグ運用の原則

- `#` プレフィックスは付けず、文字列のリストで書く（Obsidian が自動的にタグとして認識する）
- ネストタグ (`vue/transition` 等) は使わない（フラット運用で命名ブレを防ぐ）
- 本文中の `#tag` インライン記法は使わない（frontmatter に統一する）
- `worklog` は一回限りの作業ログに付ける識別タグ（純粋な技術知識と分離するため）

## マークダウンルール

`markdown-writing` skill のガイドラインに従う。重要なポイント:

- 番号付きリストは使わず `-` を使う
- 句点「。」は使わず改行で区切る
- 太字 (`**...**`) は禁止、見出しで構造化する
- 表は技術仕様の対応関係（API rename 等）に限定して使う
- 日本語と英数字の間に半角スペース

## 参考セクション

末尾に必ず「## 参考」セクションを設け、公式ドキュメントや一次情報へのリンクを列挙する。
URL は実在するものだけを記載する（推測で URL を作らない）。

## 既存 zettel との一貫性

新規作成前に `ls ~/zettel/` で既存ファイルを確認し、文体・構造を揃える。
類似トピックの zettel が既にあれば、新規作成ではなく既存ファイルへの追記を検討する。
